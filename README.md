# Axeno

A desktop messenger that uses the Signal Protocol for end-to-end encryption and Tor for transport. No accounts, no phone numbers, no email addresses. Contacts are added by exchanging one-time connection codes.

Built with [Tauri](https://tauri.app) (Rust + React) for the client and a self-hosted Rust relay server.

---

## How it works

Axeno has two components: a desktop client and a relay server.

**The relay** is a simple message broker. It holds encrypted envelopes for offline recipients and routes live delivery. It never sees plaintext and does not know who is talking to whom beyond what it can infer from transport metadata (timing, ciphertext size, source/destination mailbox IDs).

**The client** handles all encryption. Messages are encrypted with the Signal Protocol before leaving the app. The relay only forwards sealed ciphertext.

**Tor** is built into the client via [Arti](https://gitlab.torproject.org/tpo/core/arti). Traffic to .onion relay addresses goes through Tor automatically. No Tor Browser or separate Tor installation required.

### Per-contact mailboxes

Each contact gets its own dedicated receive mailbox on the relay. The relay cannot tell from the mailbox topology which of your contacts are communicating with each other. Sending goes over a fresh Tor circuit separate from the receive socket, so the relay cannot link your sending and receiving activity.

### Sealed sender

Messages use Signal's sealed-sender format. The relay sees the destination mailbox but not the sender's identity. Sender identity is proven inside the ciphertext to the recipient only.

### Encryption at rest

The local message store and identity vault are both encrypted with ChaCha20-Poly1305. The vault key is derived from your passphrase using Argon2id. The decrypted key material lives only in Rust memory and is never exposed to the frontend.

## Architecture

```
axeno/
  axeno-client/      # Tauri desktop client (source of truth)
  axeno-client2/     # Second client instance (generated by run-two-client-test.sh for local testing)
  axeno-server/      # Relay server
```

### Client stack

- **Frontend**: React + TypeScript (Vite)
- **Backend**: Rust (Tauri), Arti (Tor), libsignal-protocol
- **Key derivation**: Argon2id
- **Symmetric encryption**: ChaCha20-Poly1305
- **Transport**: WebSocket over Tor (.onion) or plaintext localhost for dev

### Relay stack

- **Rust** with Axum (WebSocket), DashMap, Tokio
- **libsignal-protocol** for issuing SenderCertificate objects
- **Persistent state**: encrypted JSON snapshot on disk

## Building

### Prerequisites

- [Rust](https://rustup.rs) (stable)
- [Node.js](https://nodejs.org) 18+
- [Tauri v2 prerequisites](https://tauri.app/start/prerequisites/) for your OS

### Client

```bash
cd axeno-client
npm install
npm run tauri dev        # dev mode
npm run tauri build      # production build
```

### Relay server

```bash
cd axeno-server
cargo run                # dev
cargo build --release    # production binary at target/release/axeno-server
```

The relay binds to `0.0.0.0:8787` by default and writes state to `./axeno-relay-data/`. Run it behind a Tor hidden service for production use.

## Local two-client test

The repo includes a script that spins up the relay and two separate client instances for end-to-end testing:

```bash
./run-two-client-test.sh
```

This starts the relay, Client A (port 1420), and Client B (port 1421) in separate terminal windows. Both clients connect to the local relay over plain WebSocket (`ws://127.0.0.1:8787`). Use the connection code flow to pair them.

## Adding contacts

Axeno does not have a directory. To start a conversation:

1. One person generates a connection code in the app (Settings or Add Contact).
2. The other person enters it.
3. The code carries Signal public identity material (identity key, signed prekey, one-time prekey). After exchange, a Signal session is established and both sides have each other's contact.

Connection codes expire and are single-use.

## Self-hosting the relay

Point the client at your relay by entering its address in the app's server settings. For real use, the relay should be exposed as a Tor hidden service so the `.onion` address is used for transport.

The relay authenticates mailbox access with per-mailbox tokens and gates sending with per-contact delivery tokens. Rotating or revoking a delivery token cuts off a specific contact's ability to send to your mailbox without affecting others.

## Security properties and limitations

| Property | Status |
|---|---|
| End-to-end encrypted (Signal Protocol) | Yes |
| Sealed sender | Yes |
| Per-contact mailbox isolation | Yes |
| Tor transport | Yes (built-in via Arti) |
| Encrypted local store | Yes (ChaCha20-Poly1305, Argon2id) |
| Forward secrecy | Yes (Signal double ratchet) |
| Post-quantum keys | Partial (Kyber prekeys supported) |
| Relay metadata visibility | Relay sees mailbox IDs, ciphertext sizes, timing |
| Anonymity | Tor reduces IP correlation; the relay is still a single point of trust for availability |

The relay is not a mixnet. It can observe when messages are sent and received and correlate them by timing even if it cannot read them. Using Tor hides your IP but does not hide the existence of communication.
